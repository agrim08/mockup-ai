# Mockup AI: Architecture Analysis & Feature Roadmap

This report provides a thorough analysis of the current state of **Mockup AI** and outlines concrete proposals to improve AI generation quality, scale the application to an enterprise-grade design platform (like Figma and Stitch), and maximize its impact for pitch decks.

---

## 1. Enhancing AI Response Quality & Visual Consistency

### Current Bottleneck Analysis
Currently, Mockup AI generates screens in two distinct stages:
1. **Config Generation (`generate-config`)**: Generates a high-level JSON outline specifying a project name, a theme, a global visual description, and descriptions of screens.
2. **Screen UI Generation (`generate-screen-ui`)**: Generates Tailwind-compatible HTML code for each screen sequentially.

**Why are screens inconsistent (e.g., massive logos, different footers)?**
* **Isolation of Context**: When the AI generates Screen 2 or Screen 3, it receives the text blueprint (`projectVisualDescription`) but **does not see the actual HTML code** generated for Screen 1. Consequently, it has to guess the HTML markup for headers, logo placements, navigation bar styles, and footers.
* **Lack of Logo & Component Constraints**: The `CODE_GENERATION_PROMPT` instructs the AI to use realistic data and Dribbble-quality styling, but doesn't define bounding limits for brand identities (e.g., logos) or exact layouts.

### Proposed Technical Solutions
To achieve pixel-perfect consistency, we should implement the following upgrades:

#### A. Code Context Lineage (Chain of Screens)
Modify the payload for `/api/generate-screen-ui` and `/api/edit-screen` to pass the code of previous screens as reference. 
* When generating Screen $N$ ($N > 1$), we inject a special section in the prompt:
  ```markdown
  ### EXISTING SCREEN REFERENCE (SCREEN 1)
  Use this HTML code to copy the exact Header, Navigation, Sidebar, Logo structure, and Footer. 
  Maintain identical container paddings, borders, and typography styles:
  [HTML Code of Screen 1]
  ```

#### B. Brand Logo & Layout Constraints in `prompt.ts`
Refine the `CODE_GENERATION_PROMPT` to restrict logo sizes and design anomalies:
```diff
  - Lists should include: avatar/logo, title, subtitle/status.
+ - Brand Logo Rules:
+     - Use inline text with a small custom icon shape inside a flex container (e.g., `flex items-center gap-2`).
+     - Keep logos compact. The logo element MUST NEVER exceed a height of `h-8` or `max-h-8`. 
+     - Never use huge full-width SVG containers or large raw images for logos.
+ - Navigation & Footer Reuse:
+     - Ensure the Header, Navigation Bar (Top/Sidebar), and Footer match the structural classes of the existing screens EXACTLY. Only modify the active states.
```

## 2. Elevating to Enterprise Level (Figma & Stitch Features)

To make Mockup AI competitive with industry-grade tools like **Figma** and **Stitch**, we can build several advanced design features:

### A. Clickable Prototyping (Inter-Screen Navigation)
Figma's killer feature is interactive prototyping. Currently, Mockup AI's buttons are static (`href="javascript:void(0)"`). We can make them interactive:
1. **Interactive Link Generation**: Instruct the AI to output custom navigation tags (e.g., `<a href="javascript:void(0)" data-navigate-to="screen-id">`) when standard navigation links are generated.
2. **Iframe Message Passing**: Inject a tiny event listener script inside the iframe via `htmlWrapper` in `data/constants.ts`:
   ```javascript
   document.addEventListener('click', (e) => {
     const target = e.target.closest('[data-navigate-to]');
     if (target) {
       window.parent.postMessage({
         type: 'NAVIGATE_TO_SCREEN',
         targetScreenId: target.getAttribute('data-navigate-to')
       }, '*');
     }
   });
   ```
3. **Canvas Auto-Focus**: In `Canvas.tsx`, listen for this message and use the `react-zoom-pan-pinch` controls (`setTransform`) to smoothly pan and zoom the viewport to center on the target screen.

### B. Dynamic Design System Engine (Inspired by Stitch)
Instead of relying on 6 static, hardcoded theme objects:
1. **Design System Schema**: Create a dynamic design system configuration table storing:
   - Primary, Secondary, Background, and Card colors.
   - Typography configurations (Google Fonts integration like Inter, Manrope, or Outfit).
   - Corner Radius settings (`ROUND_FOUR`, `ROUND_EIGHT`, `ROUND_TWELVE`, `ROUND_FULL`).
2. **Real-time Theme Swapping**: Since the HTML generated by the AI uses CSS variables (`bg-[var(--background)]`, `text-[var(--primary)]`), users can tweak their design system and see changes update **instantly across all canvas frames** in real-time without having to run expensive AI regenerations.

### C. Style Variants & AI Refinements
Build a panel to let users generate layout variants or apply specific design modifications globally (e.g., *"Convert all card elements to use glassmorphism with custom drop shadows"* or *"Convert the theme of the entire workspace to dark mode"*).

## 3. Standing Out for Idea Pitching

For founders and developers presenting their startup concepts:

### A. Fullscreen Interactive Presentation Mode
* Hide the editor sidebars, settings panel, and gridlines.
* Wrap the active screen in a realistic physical device frame (e.g., a sleek iPhone 15 frame or Safari browser shell).
* Enable the Clickable Prototyping flow so users can run a live, interactive walkthrough of their app directly during their pitch.

### B. AI Mock Data Hydrator
* Let users select a data theme (e.g., *"Fintech app in London"*, *"Luxury Real Estate in Miami"*) to instantly swap all text and numbers on the screens with contextually matching data (e.g., changing step counts to pound sterling transactions or building coordinates).

### C. Multi-Format High-Fidelity Exports
* Export the mockups as:
  - **High-res collage PNGs/PDFs** suitable for pitch deck slides.
  - **Ready-to-use codebase**: A downloadable zip containing all screens structured as Next.js components or clean HTML/CSS files.
